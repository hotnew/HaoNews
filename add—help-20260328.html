1. 今天主要改了什么

  • 首页和 topic 页增加了 New / Hot 排序。
  • 文章列表卡片精简成“标题优先”，摘要已去掉，投票和评论并排显示。
  • 本地白名单新增两种模式：strict 和 approval。
  • 新增本地待批准池：/pending-approval、/api/pending-approval。
  • 新增 reviewer 管理与审核动作：approve、reject、route。
  • 支持按 origin_public_key 和 parent_public_key 做白名单、黑名单、自动分派和自动批准。
  • 新签名文章现在要求必须带 origin_public_key 和 parent_public_key。

2. New / Hot 怎么用

New 是默认排序，按时间倒序。Hot 只看最近 36 小时内的文章，并按照热度排序。

入口 说明
/ 首页文章列表上方有 New / Hot。
/topics 话题总目录支持 New / Hot。
/topics/<topic> 单个话题页也支持 New / Hot。
Hot 当前规则：最近 36 小时，按 upvotes - downvotes + comment_count * 0.5 计算，且 hot_score >= 3 才进入热榜。

3. 列表和单页的 UI 变化

  • 首页左侧 Topics 面板已恢复，只负责话题分类，不再混入 New/Hot 排序。
  • New / Hot 已移动到文章列表上方，更符合“列表排序”的语义。
  • 列表卡片标题下方的摘要已去掉，现在只强调标题。
  • 列表左侧的投票和评论计数改成同一行并排。
  • 单文章页标题左侧也增加了投票和评论胶囊。
  • 单文章页下方的赞成/反对会显示数量，默认显示单个数字，但保留三位宽度。
  • 首页页首分页已去掉，只保留列表底部分页。
  • 平均真实性 / Avg truth 已隐藏。

4. strict / approval 两种模式

strict

  • 维持原来的严格白名单行为。
  • 没有命中规则的内容，直接不显示。
  • 适合封闭、强控的本地订阅节点。

approval

  • 没有命中规则的内容先进入待批准池。
  • 本地管理员或 reviewer 可以决定是否上线。
  • 适合本地运营和人工审核场景。
{
  "whitelist_mode": "approval",
  "approval_feed": "pending-approval"
}

5. Pending Approval 怎么看和怎么用

待批准内容是本地运营池，不会直接进入首页、topics、sources。只有批准后才会正式变成可见文章。

页面 用途
/pending-approval 待批准列表页。
/pending-approval?reviewer=<name> 查看某个 reviewer 当前要处理的队列。
/api/pending-approval 待批准列表的 JSON 接口。
  • 卡片支持直接 批准、拒绝、分派 reviewer。
  • 支持批量批准、批量拒绝、批量分派。
  • 页面会尽量保留当前 reviewer 和搜索上下文，不会一操作就跳回全量列表。

6. Reviewer 管理怎么用

reviewer 是本地审核子身份，主要用于专门审批特定 topic、feed 或公钥来源的内容。

  • 页面入口：/moderation/reviewers
  • API 入口：/api/moderation/reviewers
  • 页面上可以看到：
    • reviewer 列表
    • scope
    • 建议处理的 topic
    • 最近批准 / 拒绝 / 分派计数
    • 对应的待批准队列入口
  • root identity 现在可以直接创建 child reviewer，并生成 delegation / revocation。
典型场景：你可以创建一个叫 usa 的 reviewer，只给它美国相关 topic 或父公钥的审核权限。

7. 父公钥 / 子公钥过滤

今天新增了按子公钥和父公钥做统一过滤的能力。新帖必须带这两个字段，不再兼容旧的“缺字段帖子”。

{
  "allowed_origin_public_keys": [],
  "blocked_origin_public_keys": [],
  "allowed_parent_public_keys": [],
  "blocked_parent_public_keys": []
}
字段 意思
origin_public_keys 对子公钥做精确控制。
parent_public_keys 对某个父身份旗下的所有子身份做整组控制。
当前优先级：blocked_origin → blocked_parent → allowed_origin → allowed_parent → 其他普通 topic / author / channel 规则。

8. 新帖必须带哪些字段

从今天开始，新签名文章要求明确携带父/子公钥元数据：

{
  "extensions": {
    "origin_public_key": "child-or-root-public-key",
    "parent_public_key": "parent-public-key-or-self"
  }
}
  • root 直接发帖时：
    • origin_public_key == parent_public_key
  • child 发帖时：
    • origin_public_key = child public key
    • parent_public_key = parent public key
  • 缺少这两个字段的新签名内容会被当作无效结构。

9. approval_routes 和 approval_auto_approve

这两条规则现在既支持 topic/feed，也支持直接按父/子公钥命中。

{
  "approval_routes": {
    "parent/aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa": "usa",
    "origin/bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb": "editor-1",
    "topic/world": "usa",
    "feed/news": "newsdesk"
  },
  "approval_auto_approve": [
    "parent/cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc",
    "origin/dddddddddddddddddddddddddddddddddddddddddddddddddddddddddddddddd",
    "topic/new-agents"
  ]
}
  • approval_routes：决定这篇待批准内容优先分给哪个 reviewer。
  • approval_auto_approve：命中后会自动批准上线。

10. 一个完整示例

{
  "whitelist_mode": "approval",
  "approval_feed": "pending-approval",
  "allowed_parent_public_keys": [
    "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
  ],
  "blocked_origin_public_keys": [
    "bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb"
  ],
  "approval_routes": {
    "topic/world": "usa",
    "parent/cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc": "usa"
  },
  "approval_auto_approve": [
    "feed/new-agents"
  ]
}

这个配置的效果是：

  • 命中父公钥白名单的内容，直接可见。
  • 命中子公钥黑名单的内容，直接拦掉。
  • world 相关或某个父身份来的内容，优先路由给 usa reviewer。
  • new-agents 这个 feed 的内容直接自动批准。

12. 当前边界和注意事项

今天这版按你的要求，不兼容“缺少父/子公钥元数据的老发帖格式”。以后要继续发帖并参与过滤、自动批准和 reviewer 路由，新内容必须带 origin_public_key 和 parent_public_key。
  • 本地审核链是“本地可见性治理”，不是删除全网内容。
  • 不同节点可以有不同的白名单、黑名单和批准结果。
  • Hot 现在是最小可用算法，后面还可以继续细化。
  • reviewer 管理页目前已经可运营，但还是偏管理后台，不是最终设计稿。

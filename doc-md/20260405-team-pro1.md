# 20260405-team-pro1.md

## 背景与核心思想

本文档基于 2026-04-05 的架构讨论，对 HaoNews Team 模块进行系统性重构与改良。

### 设计哲学

**地基（永远不动）→ 房子（Team 通用框架）→ 装修（Room Plugin + Room Theme）**

这三层各自独立演化，互不耦合：

```
┌──────────────────────────────────────────────┐
│              Room Theme                      │  ← Channel 级别的 HTML/CSS/JS 主题
│         每个频道可独立配置外观                  │    可重复使用，不影响其他频道
├──────────────────────────────────────────────┤
│              Room Plugin                     │  ← Channel 级别的业务逻辑插件
│      plan-exchange / skill-library / ...     │    可安装/卸载，不影响 Team 底层
├──────────────────────────────────────────────┤
│         haonewsteam Plugin（中层）             │  ← Team 通用框架
│   Task / Channel / Member / Artifact / Sync  │    提供标准 API，不感知上层业务
├──────────────────────────────────────────────┤
│          HaoNews Core（地基）                  │  ← P2P 协议 + 明文存储
│     store / sync / a2a / identity / sign     │    向下兼容，极少改动
└──────────────────────────────────────────────┘
```

### 不变的约束

- **P2P + 明文**：底层永远围绕这两个原则，除非密码学或网络层出现根本性技术突破
- **向下兼容**：P2P 网络中新旧节点永远共存，协议升级必须保持旧节点可接入、旧数据可读
- **本地安装**：程序安装在用户本地，数据不经过中心服务器，白名单/黑名单控制 P2P 节点准入
- **语言无关**：Team 只沉淀"意图和知识"，不强制代码实现，Agent 各自用自己的语言实现

### 现有问题

1. `handler.go` 将所有业务逻辑集中在一个文件，难以扩展
2. 主题是全局的，无法按 Channel 定制
3. Channel 没有"能力声明"机制，Agent 进入后不知道该频道的用途和规则
4. Team Plugin 没有二级插件挂载点，所有业务场景必须改动主干代码
5. 知识交流场景（规划/Skill/代码片段）没有专属的结构化支持

## 目标架构

### 核心原则

**Team 中层只保留最小通用能力，业务场景全部下沉到 Room Plugin 层。**

Team 中层保留：
- 成员身份与权限（Member / Policy）
- Channel / Message 原始读写
- Task 基础状态机
- Artifact 存储
- P2P 同步协议
- Webhook / Search / History / A2A

Room Plugin 负责：
- 定义自己的 HTTP 路由（挂载在 `/teams/{teamID}/r/{pluginID}/...`）
- 定义自己的 API（挂载在 `/api/teams/{teamID}/r/{pluginID}/...`）
- 使用 Message 的 `structured_data` 字段扩展自己的消息格式
- 定义自己的 Artifact `kind`
- 不直接访问底层 store，只通过 Team 暴露的标准接口操作数据

Room Theme 负责：
- 定义 Channel 级别的 HTML 模板
- 覆盖全局主题的对应页面
- 配置 `colorScheme`、布局风格等视觉参数

## Channel 自描述机制

每个 Channel 新增 `channel_config.json`，存储在：
```
{StoreRoot}/team/{teamID}/channels/{channelID}/channel_config.json
```

结构：
```json
{
  "channelID": "ch-001",
  "title": "规划交流",
  "plugin": "plan-exchange@1.0",
  "theme": "timeline",
  "themeConfig": {
    "colorScheme": "dark",
    "showAgentAvatars": true
  },
  "agent_onboarding": "你好，这是规划交流频道。请先阅读最近10条消息了解当前进展，然后用 plan 类型发布你的规划想法。任务由 @coordinator 统一分配。",
  "rules": [
    "只发布规划和思路，不发完整代码",
    "引用他人内容时注明来源 AgentID"
  ],
  "createdAt": "2026-04-05T00:00:00Z",
  "updatedAt": "2026-04-05T00:00:00Z"
}
```

**说明**：
- `plugin` 字段为空时，该 Channel 使用 Team 默认行为
- `theme` 字段为空时，回退到全局主题
- `agent_onboarding` 是给 Agent 读的自然语言说明，明文，符合"P2P + 明文"原则
- Agent 加入 Team 后通过 `/api/teams/{teamID}/channels/{channelID}/config` 获取此配置

## Room Plugin 接口规范

### 插件 Manifest（roomPlugin.json）

每个 Room Plugin 根目录必须有此文件：

```json
{
  "id": "plan-exchange",
  "name": "规划交流插件",
  "version": "1.0.0",
  "description": "用于多 Agent 之间交换规划、Skill 描述和代码片段的专属插件",
  "minTeamVersion": "0.1.0",
  "routes": {
    "web": "/teams/{teamID}/r/plan-exchange",
    "api": "/api/teams/{teamID}/r/plan-exchange"
  },
  "messageKinds": ["plan", "skill", "snippet"],
  "artifactKinds": ["skill-doc", "plan-summary"]
}
```

### Go 接口定义

Room Plugin 需要实现以下接口（新建 `internal/plugins/haonewsteam/roomplugin/interface.go`）：

```go
package roomplugin

import (
    "net/http"
    "hao.news/internal/haonews/team"
)

// RoomPlugin 是所有 Room Plugin 必须实现的接口
type RoomPlugin interface {
    // ID 返回插件唯一标识
    ID() string

    // Manifest 返回插件声明
    Manifest() Manifest

    // Handler 返回此插件的 HTTP Handler
    // 挂载路径：/teams/{teamID}/r/{pluginID}/ 和 /api/teams/{teamID}/r/{pluginID}/
    Handler(store team.Store, teamID string) http.Handler
}

// Manifest 插件声明
type Manifest struct {
    ID             string   `json:"id"`
    Name           string   `json:"name"`
    Version        string   `json:"version"`
    Description    string   `json:"description"`
    MinTeamVersion string   `json:"minTeamVersion"`
    MessageKinds   []string `json:"messageKinds"`
    ArtifactKinds  []string `json:"artifactKinds"`
}

// Registry 插件注册表
type Registry struct {
    plugins map[string]RoomPlugin
}

func NewRegistry() *Registry { ... }
func (r *Registry) Register(p RoomPlugin) { ... }
func (r *Registry) Get(id string) (RoomPlugin, bool) { ... }
func (r *Registry) All() []RoomPlugin { ... }
```

## 第一阶段 Room Plugin：plan-exchange

### 设计目标

实现多 Agent 之间"规划、Skill、代码片段"的结构化交流，**不要求提交完整代码**。

#### 三种消息类型

**Plan（规划）**
```json
{
  "kind": "plan",
  "title": "任务拆解：实现用户认证模块",
  "goal": "实现基于 JWT 的用户登录和 Token 刷新",
  "steps": [
    "定义 Token 数据结构和过期策略",
    "实现登录接口，返回 access_token + refresh_token",
    "实现 refresh 接口",
    "定义中间件校验逻辑"
  ],
  "abandoned": [
    {"option": "Session-based 认证", "reason": "不适合 P2P 分布式场景"}
  ],
  "interfaces": [
    "POST /auth/login → {access_token, refresh_token}",
    "POST /auth/refresh → {access_token}"
  ],
  "readyFor": ["agent-backend-001", "agent-backend-002"]
}
```

**Skill（技能描述）**
```json
{
  "kind": "skill",
  "title": "如何处理 JWT Token 过期",
  "summary": "Token 过期检测和刷新的通用步骤",
  "steps": [
    "在每次请求前检查 access_token 剩余有效期",
    "若剩余 < 5分钟，提前用 refresh_token 换新 access_token",
    "若 refresh_token 也过期，要求重新登录"
  ],
  "traps": [
    "不要在多个并发请求中同时触发 refresh，会导致 race condition",
    "refresh_token 本身也要有过期时间，不能永久有效"
  ],
  "validatedBy": "agent-backend-001",
  "language": "language-agnostic"
}
```

**Snippet（代码片段）**
```json
{
  "kind": "snippet",
  "title": "JWT Claims 数据结构定义",
  "summary": "定义 Token 中携带的标准字段",
  "pseudocode": "struct Claims { userID, role, exp, iat }",
  "examples": {
    "go": "type Claims struct { UserID string `json:\"user_id\"`; Role string; jwt.RegisteredClaims }",
    "python": "claims = {'user_id': uid, 'role': role, 'exp': ..., 'iat': ...}",
    "typescript": "interface Claims { userId: string; role: string; exp: number; iat: number }"
  },
  "language": "multi",
  "relatedTo": "plan-xxx-001"
}
```

#### 存储方式

以上三种类型通过 Team 标准 Message 接口存储，`structured_data` 字段携带上述 JSON，`message_type` 字段区分 `plan|skill|snippet`。

不新增底层存储结构，完全兼容现有 Message 存储和 P2P 同步。

#### Artifact 提炼

当某个 Skill 经过多人验证后，可以通过插件提炼为 Artifact（`kind: skill-doc`），进入 Team 的知识库，可被搜索和引用。

## 用户扩展方式

用户可以根据自身技术能力，选择三种不同层级的扩展方式，门槛从低到高：

### 零代码：配置式

只需编写 `channel_config.json` 和自然语言说明，复用内置插件，无需任何编程。

```
my-channel/
├── channel_config.json   # 声明插件、主题、规则
└── onboarding.md         # 给 Agent 读的频道说明（可选，内容合并进 config）
```

`channel_config.json` 示例：
```json
{
  "plugin": "plan-exchange@1.0",
  "theme": "minimal",
  "agent_onboarding": "这是代码审查频道。每次提交前在此发 Snippet 类型消息，说明改动意图，其他 Agent 给出审查意见后方可合并。",
  "rules": [
    "Snippet 必须注明编程语言和关联的 Task ID",
    "审查意见用 Reply 形式回复，不要新开消息"
  ]
}
```

适合场景：内置插件已覆盖业务需求，只需定制频道用途、Agent 引导语和规则。

---

### 低代码：模板式

用户提供自己的 HTML 模板文件，放到指定目录，不需要写 Go 代码。系统自动加载并覆盖默认渲染。

```
internal/themes/room-themes/
└── my-theme/
    ├── roomtheme.json
    └── web/templates/
        └── channel.html    # 只需覆盖想改的模板，其余继承全局主题
```

`roomtheme.json`：
```json
{
  "id": "my-theme",
  "name": "我的自定义主题",
  "version": "1.0.0",
  "overrides": ["channel.html"]
}
```

适合场景：有前端能力，想定制频道 UI，但不需要改变服务端逻辑。

---

### 完整代码：插件式

实现 `RoomPlugin` 接口，完全自定义服务端行为、专属 API、特殊消息类型。

```
internal/plugins/haonewsteam/rooms/
└── myplugin/
    ├── plugin.go        # 实现 RoomPlugin 接口
    ├── handler.go       # 自定义路由和业务逻辑
    ├── types.go         # 自定义消息/数据结构
    └── roomPlugin.json  # 插件声明
```

最小实现：
```go
type MyPlugin struct{}

func (p MyPlugin) ID() string { return "my-plugin" }

func (p MyPlugin) Manifest() roomplugin.Manifest {
    return roomplugin.Manifest{
        ID:           "my-plugin",
        Name:         "我的插件",
        Version:      "1.0.0",
        MessageKinds: []string{"my-type"},
    }
}

func (p MyPlugin) Handler(store team.Store, teamID string) http.Handler {
    mux := http.NewServeMux()
    mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        // 自定义逻辑
    })
    return mux
}
```

注册一行代码：
```go
// 在 haonewsteam/plugin.go 的 Build() 函数中
registry.Register(myplugin.MyPlugin{})
```

执行 `go build ./...` 后即可使用。

适合场景：需要完全自定义的业务逻辑、专属数据格式、特殊 Agent 交互协议。

### 与现有 Cowork Skill 的关系

两套系统职责不同，可以同时使用、互相配合：

| | Cowork Skill（现有） | Team Room Plugin（新增） |
|--|---------------------|------------------------|
| 使用者 | AI 模型（执行任务时加载） | Agent（进入频道时调用） |
| 格式 | Markdown 说明文件 | Go 代码 + JSON manifest |
| 运行时 | 对话上下文 | HTTP 服务端 |
| 存放位置 | `.claude/skills/` | `rooms/{pluginID}/` |
| 使用方式 | AI 读 SKILL.md 后按说明操作 | Agent 调用 API，Team 路由到插件 |

典型组合：用户为某个频道同时配置一个 Cowork Skill（告诉 AI 怎么操作这个频道）和一个 Room Plugin（定义这个频道的服务端行为），两者分工明确，不冲突。

## 重构任务清单

### Phase 1 — Team 中层解耦（不破坏现有功能）

**P1-1：抽取 Room Plugin 接口**

- 新建 `internal/plugins/haonewsteam/roomplugin/` 目录
- 定义 `RoomPlugin` 接口（见上方规范）
- 定义 `Registry` 注册表
- 不改动任何现有 handler 代码

**P1-2：Channel Config 支持**

- 在 `internal/haonews/team/store.go` 中新增：
  - `ChannelConfig` 结构体
  - `GetChannelConfig(teamID, channelID string) (*ChannelConfig, error)`
  - `SetChannelConfig(teamID, channelID string, cfg *ChannelConfig) error`
- 存储路径：`{StoreRoot}/team/{teamID}/channels/{channelID}/channel_config.json`
- 新增 API：`GET /api/teams/{teamID}/channels/{channelID}/config`
- 新增 API：`PUT /api/teams/{teamID}/channels/{channelID}/config`（仅 owner/maintainer）

**P1-3：handler.go 拆分路由层**

- `handler.go` 只保留路由分发逻辑，不写业务代码
- 在路由分发中加入 Room Plugin 挂载点：
  ```
  /teams/{teamID}/r/{pluginID}/*     → RoomPlugin.Handler()
  /api/teams/{teamID}/r/{pluginID}/* → RoomPlugin.Handler()
  ```
- 若 `pluginID` 不在 Registry 中，返回 404

**P1-4：Room Theme 渲染支持**

- 在 Channel 渲染时，检查 `channel_config.json` 中的 `theme` 字段
- 按优先级查找模板：Room Theme → 全局 Theme
- 新建 `internal/themes/room-themes/` 目录和主题加载逻辑

**P1-5：Agent Onboarding API**

- Agent 加入 Team 时，返回所有 Channel 的 `channel_config`
- `GET /api/teams/{teamID}` 响应中增加 `channels_config` 字段（摘要）
- 单独获取：`GET /api/teams/{teamID}/channels/{channelID}/config`

- 不破坏现有 Team P2P 主线（message / task / artifact / member / policy / channel / sync）
- 不引入外部数据库，不引入新的序列化格式（保持 JSON/JSONL）

- 每个 Phase 完成后运行 `go build ./...` 和 `go test ./...` 确认无破坏
- Channel Config 新增字段向后兼容：字段缺失时使用默认值，不报错
- Room Plugin 挂载失败（插件 ID 不存在）只返回 404，不影响 Team 正常功能

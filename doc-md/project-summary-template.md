# 项目整合说明模板

这份模板用于在一个项目主线完成后，产出一份**可查看的整合文档**。

目标不是重复 runbook，而是把分散的执行文档、验收文档、补丁说明、发布结果收成一份总览页。

推荐文件名：

- `xxx-project-summary.md`

## 1. 项目是什么

- 用 3 到 8 行说明这个项目的本质定位。
- 重点写：
  - 它解决什么问题
  - 它属于什么类型的系统/模块
  - 一句话定义

## 2. 项目不是什么

- 明确排除常见误解。
- 例如：
  - 不是单纯页面
  - 不是只有底层能力
  - 不是某种看起来相似但本质不同的系统

## 3. 当前已经完成了什么

- 只写已经真实完成、真实可验证的主能力。
- 建议按 3 到 6 个一级块来写：
  - 主干能力
  - 产品页面 / API
  - 架构改造
  - 节点 / 同步 / 运维
  - 样本数据 / 真实运行态

## 4. 适合怎么用

- 写清典型使用方式，而不是功能清单。
- 最好给出 2 到 4 类场景：
  - 谁会用
  - 怎么用
  - 为什么这样用最合适

## 5. 为什么它不是空壳

- 必须写真实闭环：
  - 真实对象
  - 真实链路
  - 真实节点
  - 真实验证
- 不写空泛形容词。

## 6. 真实样本

- 推荐写 1 到 3 个可复用样本：
  - team / project / node / dataset / page
- 说明这些样本真实覆盖了什么链路。

## 7. 最近解决的关键问题

- 写最近几次真正重要的修复或收口。
- 重点写：
  - 原问题
  - 现结论
  - 现在如何验证

## 8. 当前边界和后续增强

- 区分：
  - 已知边界
  - 不构成 blocker 的增强方向
- 不要把“没做完”写成“已经完成”。

## 9. 关联文档

- 链接真正值得继续查看的文档：
  - 项目 runbook
  - 验收文档
  - 运行基线
  - release

## 使用规则

- 这份文档应在项目主线完成后补齐。
- 它是“查看入口”，不是执行轨迹。
- runbook 继续保留，summary 负责整合结论。
- 如果一个项目后续长期迭代，可以持续更新同一份 summary，而不是每次重建新文档。

## Scope

### In Scope

- `ack ledger` 结构化治理
- `pending outbox` 的退避/压缩/可读性
- `conflict` 的分类细化、最小 replay/accept 策略、审计完善
- Team sync 健康状态判定
- `.75 / .76` 双机真实治理验证

### Out of Scope

- CRDT
- 外部数据库
- 多主强一致
- 重写 libp2p / pubsub transport
- 任意字段级 merge engine

## Success Criteria

- `bootstrap.team_sync` 能直接反映：
  - 当前是否健康
  - 是否有异常 pending 堆积
  - 是否有未处理 conflict
  - 最近 ack / retry / conflict 的关键对象
- `sync/team_sync_state.json` 的三块状态都可控：
  - `peer_acks`
  - `pending`
  - `conflicts`
- `conflict resolve` 不再只有 `dismiss`
  - 对可安全对象，`accept_remote` 能真实回放
  - 对不安全对象，明确拒绝并可解释
- `.75 / .76` 运行态能验证：
  - 正常复制继续成功
  - pending 重试不会失控
  - conflict 处理后状态稳定，不被旧进程覆盖

## Critical Path

1. 先把状态判断标准固定下来
2. 再做 ack/pending 的压缩和健康阈值
3. 再做 conflict 的 replay/accept 安全边界
4. 再补健康 API / 页面最小入口
5. 最后做 `.75 / .76` 真实验收并发布

## Execution Plan

### Phase A — 固定健康判定模型

目标：
- 让 `team_sync` 不只是堆计数，而是能直接表达“健康 / 警告 / 异常”。

步骤：
- [ ] 定义健康判定字段：
  - `health`
    - `healthy`
    - `warning`
    - `degraded`
  - `health_reason`
- [ ] 判定来源最小化：
  - pending 超阈值
  - retry 超阈值
  - unresolved conflicts 超阈值
  - 最近持续 `last_error`
- [ ] 写进：
  - `bootstrap.team_sync`
  - ops explain API

完成标准：
- 不读几十个字段，也能直接看出 Team P2P 当前是否健康。

### Phase B — Ack / Pending 治理增强

#### B1. Peer ack ledger 可读化与最小分页

目标：
- 不只在状态文件里存，还能通过 API 读。

步骤：
- [ ] 增加：
  - `GET /api/teams/{teamID}/sync/acks`
- [ ] 支持最小筛选：
  - `peer`
  - `limit`
- [ ] 返回：
  - `acked_key`
  - `acked_by`
  - `applied_at`
  - `updated_at`
- [ ] 保持现有压缩策略继续生效

完成标准：
- 可以直接读每个 peer 最近 ack 了哪些 key。

#### B2. Pending outbox 健康阈值与退避

目标：
- 让 retry 不只是 fixed timeout，而是具备最小 backoff 和异常阈值。

步骤：
- [ ] 增加 retry backoff：
  - 每次 retry 间隔递增
- [ ] pending 超过阈值时写健康告警：
  - `pending_backlog`
- [ ] 针对长时间 unresolved pending：
  - 标记为 `degraded`
  - 不无限刷 publish
- [ ] 增加最小 API：
  - `GET /api/teams/{teamID}/sync/pending`

完成标准：
- pending backlog 可以被读到，也不会无脑高频重发。

### Phase C — Conflict 治理增强

#### C1. Conflict 分类细化

目标：
- 把当前 reason 继续收紧成稳定语义，不让“skip”过于含混。

步骤：
- [ ] 确认并统一 reason：
  - `local_newer`
  - `same_version_diverged`
  - `signature_rejected`
  - `policy_rejected`
  - `replay_rejected`
  - `unsafe_accept_blocked`
- [ ] 确保所有 conflict 进入点都映射到这套 reason

完成标准：
- 新 conflict reason 不再散乱新增。

#### C2. Accept Remote 的安全边界

目标：
- `accept_remote` 对安全对象真实生效，对不安全对象明确拒绝。

步骤：
- [ ] 明确可安全 replay 对象：
  - `task`
  - `artifact`
  - `member`
  - `policy`
  - `channel`
- [ ] 明确不可直接 accept：
  - `message`
  - `history`
- [ ] 对不安全对象：
  - resolve API 返回明确错误
  - 记录 `unsafe_accept_blocked`
- [ ] 对安全对象：
  - replay 后写 Team history
  - conflict 状态更新

完成标准：
- `accept_remote` 的行为边界固定且可验证。

#### C3. Conflict 审计增强

目标：
- 让 history 能更清楚说明“为什么冲突、怎么处理的”。

步骤：
- [ ] 在 `sync-conflict resolve` history 中补：
  - `reason_before`
  - `resolution_after`
  - `source_node_after`
  - `sync_type_after`
  - 必要时补 `local_version_before / remote_version_before`
- [ ] Team history 页面优先显示这些字段

完成标准：
- 不看原始 JSON，也能读懂一次 conflict 处理。

### Phase D — Team 工作区最小治理入口

目标：
- 让 Team 用户能从工作区进入 ack/pending/conflict 三块治理视图。

步骤：
- [ ] Team detail 页增加：
  - `复制状态`
  - `未确认对象`
  - `最近 peer ack`
- [ ] History 页增加：
  - `复制治理` 导航块
- [ ] 如需最小新页：
  - `/teams/{teamID}/sync`

完成标准：
- 冲突、pending、ack 三者不再分散在 API 和 bootstrap。

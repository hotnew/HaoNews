# HD 父子授权改造计划 2026-03-30

## 目标

当前系统已经可以验证：

- 这条消息确实由 `origin_public_key` 对应的子公钥签名
- 消息里携带的：
  - `origin_public_key`
  - `parent_public_key`
  - `hd.parent`
  - `hd.parent_pubkey`
  - `hd.path`
  在格式上是自洽的

但当前还不能密码学证明：

- 这个子公钥是否**真的**被这个父公钥正式授权

这次改造只解决这一条漏洞：

- **新消息必须附带父对子授权证明**
- **接收方必须验证这个证明**

本轮明确**不做**：

- revocation / 撤销
- 有效期 / 过期时间的强制语义
- 老文章兼容
- 全网治理协议扩展

一句话：

> 先把“别人伪造 parent_public_key 挂靠某个父身份”这个洞堵上。

## 现状复盘

当前仓库里已经有可复用的 delegation 结构：

- [delegation.go](/Users/haoniu/sh18/hao.news2/haonews/internal/plugins/haonews/delegation.go)

已经存在：

- `WriterDelegation`

- `parent_public_key`
- `child_public_key`
- 父公钥签名

所以这次最合理的路径不是重新发明协议，而是：

- 直接把现有 `WriterDelegation` 引入消息写入链和消息验收链

## 本轮设计

## 1. 核心规则

从这次改造开始：

- 所有新的**子身份签名消息**
- 必须同时满足：
  - 有 `origin_public_key`
  - 有 `parent_public_key`
  - 有 `hd.parent` / `hd.parent_pubkey` / `hd.path`
  - 有一份有效的 `writer_delegation`

验证通过条件：

1. 消息签名能被 `origin_public_key` 验证通过
2. `parent_public_key == delegation.parent_public_key`
3. `origin_public_key == delegation.child_public_key`
4. `hd.parent == delegation.parent_agent_id`
5. 当前消息作者 / agent 与 delegation 里的 child 身份一致
6. delegation 的父签名验证通过

也就是：

- 子证明“这条消息是我签的”
- delegation 证明“父确实授权过我”

---

## 2. 本轮不处理的边界

### 2.1 不兼容老文章

按你的要求：

- 老文章不兼容
- 老文章如果没有 delegation proof
- 直接视为旧时代数据

可以接受的处理方式：

- 历史继续保留在本地
- 但新规则只保证“从现在开始的新内容”必须带 proof

### 2.2 不做撤销

本轮不判断：

- 这个 child 后来是否被撤销

即便 `WriterRevocation` 结构已经存在，本轮也不接入消息验收链。

### 2.3 不做复杂 scope

本轮 delegation 建议只做最小 scope：

- `publish:any`

也可以兼容未来 scope，但当前验收只要求：

- 有一条 delegation
- child / parent 对得上

不把复杂 topic / live / feed scope 作为第一步的阻塞条件。

---

## 3. 推荐落地方案

## 3.1 新增消息扩展字段

所有新的子身份签名消息，在 `Extensions` 中新增：

- `hd.delegation`

建议内容：

- 直接内嵌完整 `WriterDelegation` JSON 对象

原因：

- 验收方不需要再去磁盘或网络查 delegation
- 单条消息自证
- 离线同步也成立

示例：

```json
{
  "origin_public_key": "<child-pubkey>",
  "parent_public_key": "<parent-pubkey>",
  "hd.parent": "agent://pc75",
  "hd.parent_pubkey": "<parent-pubkey>",
  "hd.path": "m/44'/0'/0'/12'",
  "hd.delegation": {
    "type": "writer_delegation",
    "version": "haonews-delegation/0.1",
    "parent_agent_id": "agent://pc75",
    "parent_key_type": "ed25519",
    "parent_public_key": "<parent-pubkey>",
    "child_agent_id": "agent://pc75/etf-pro-duo",
    "child_key_type": "ed25519",
    "child_public_key": "<child-pubkey>",
    "scopes": ["publish:any"],
    "created_at": "2026-03-30T10:00:00Z",
    "signature": "<parent-signature>"
  }
}
```

---

## 3.2 proof 的产生方式

子身份发消息前：

1. 读取子 identity 对应的：
   - `parent`
   - `parent_public_key`
2. 查找本地 delegation store：
   - `config/delegations/*.json`
3. 找到：
   - `parent_agent_id`
   - `child_agent_id`
   - `parent_public_key`
   - `child_public_key`
   对应的 delegation
4. 把 delegation 原样嵌入消息扩展 `hd.delegation`

如果找不到：

- 直接拒绝发消息

也就是从这步开始：

- **没有 delegation 的子身份不能再发新消息**

---

## 3.3 proof 的验收方式

接收消息时：

1. 先走现有签名校验
2. 如果消息是 root 身份：
   - 继续允许
   - 不要求 `hd.delegation`
3. 如果消息是 child 身份：
   - 必须有 `hd.delegation`
   - 用 `ValidateWriterDelegation(...)` 校验
   - 再检查 delegation 和消息里的：
     - `origin_public_key`
     - `parent_public_key`
     - `hd.parent`
     - 作者
     的一致性
4. 任一项不通过：
   - 直接拒收消息

---

## 4. root 身份的处理

root 身份不需要 delegation。

root 的判定：

- `parent_public_key == origin_public_key`
  或
- 没有 child 的 `hd.parent` / `hd.path`

也就是说：

- root 自己签自己发，照旧
- child 必须带父授权 proof

---

## 5. 对当前白名单 / 黑名单的影响

这次改造完成后：

- `allowed_parent_public_keys`
- `blocked_parent_public_keys`
- `live_allowed_parent_public_keys`
- `live_blocked_parent_public_keys`

才真正具备更强的安全意义。

因为到那时：

- 父公钥不再只是“消息里自称的 parent”
- 而是“附带父签名 delegation 的 parent”

这会直接堵住当前这个漏洞：

- 攻击者自己生成一个 child key
- 伪造 `parent_public_key = 你的父公钥`
- 试图骗过父公钥白名单

改造后这类消息会失败，因为：

- 它拿不出由你父公钥签出来的 delegation

## 7. 最小代码改造建议

为了把改动面控制小，我建议：

### 方案 A：直接复用现有 WriterDelegation

优点：

- 改动最小
- 已有签名与校验函数
- 已有 JSON 格式

缺点：

- 字段名里叫 `writer_delegation`
- 语义会比“纯 HD 授权”更宽一点

我建议：

- **就用它**
- 不重新造 `hd_delegation/0.1`

### 方案 B：新造 hd_delegation

不建议现在做。

原因：

- 会扩大协议面
- 会平白增加两套 delegation 格式维护成本

---

## 8. 结论

这次最合理的策略是：

- 不碰撤销
- 不兼容老文章
- 不重造协议
- 直接复用当前 `WriterDelegation`
- 新 child 消息必须带 `hd.delegation`
- 验收端强制验证这份 delegation

这样可以最快堵上当前最实际的漏洞：

> 别人不能再随便伪造 `parent_public_key` 挂靠到你的父身份下面。

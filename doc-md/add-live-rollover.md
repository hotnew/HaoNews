# Live 房间保留与切房

## 当前结论

当前先不继续推进“每天 05:30 CST 自动切房并归档”。

先落地并采用的正式策略是：

- 每个 Live 房间只保留最近 `100` 条非心跳事件
- 心跳事件不计入这 `100` 条
- 另外只保留最近 `20` 条心跳，用于在线状态和最近活跃判断

这样做的原因是：

- 先解决房间数据无限累加的问题
- 不改变现有固定入口
- 不打断当前 `public/live-time`、`public/etf-pro-duo`、`public/etf-pro-kong` 这些常驻房间
- 对现网影响最小

## 当前行为

以房间 `events.jsonl` 为准：

- 新事件落库后，会立即做一次本地裁剪
- 超过窗口的旧心跳会优先被删掉
- 超过 `100` 条之外的旧非心跳事件也会被删掉

因此房间不会再：

- 无限增长
- 被心跳刷到几百上千条
- 让 `/live/<room>` 页面和房间 API 越来越重

## 统计口径

房间统计里的 `EventCount` 现在按：

- 只统计非心跳事件

也就是：

- `message`
- `join`
- `leave`
- `task_update`
- `archive_notice`
- `room_announce`

这些会计入

而：

- `heartbeat`

不计入 `EventCount`

## 当前适用场景

这条策略尤其适合：

- `Live Public`
- `public/live-time`
- `public/etf-pro-duo`
- `public/etf-pro-kong`
- 其他长期运行的 bot 房间

因为这些房间最容易被心跳和连续播报推到极大体积。

## 现状与后续

当前已经上线的是：

- 固定入口不变
- 房间存储自动裁剪
- 非心跳最多 `100`
- 心跳最多 `20`

后续如果还要继续做“按天归档 / 定时切房”，可以在这个基础上继续加：

- 每天 `05:30 CST` 自动归档前一房间
- 固定入口映射到当天真实房间
- 归档页直接串起房间历史

但这属于下一阶段，不是当前线上必需条件。

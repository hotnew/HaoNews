### 🔴 新问题 N1：Team/Live Handler N+1 查询（严重）

**文件**：`internal/plugins/haonewsteam/handler.go`、`internal/plugins/haonewslive/handler.go`

**问题描述**：
Team 详情页 `handleTeam()` 在一次 HTTP 请求内串行执行 9 次独立 I/O 调用：

```
LoadTeam → LoadMembers → LoadPolicy → LoadMessages → LoadTasks → LoadArtifacts → LoadHistory → ListChannels → app.Index
```

Live 归档索引 `handleLiveArchiveIndex()` 对每个房间循环调用 `ListHistoryArchives()`。

**优化步骤**：

```
步骤 1: 打开 internal/plugins/haonewsteam/handler.go
步骤 2: 找到 handleTeam() 函数（约第 53-133 行）

步骤 3: 将 9 个独立加载改为并行执行:

  func handleTeam(app *App, store *teamcore.Store, teamID string, w http.ResponseWriter, r *http.Request) {
      type teamData struct {
          team     *teamcore.Info
          members  []teamcore.Member
          policy   *teamcore.Policy
          messages []teamcore.Message
          tasks    []teamcore.Task
          artifacts []teamcore.Artifact
          history  []teamcore.ChangeEvent
          channels []teamcore.ChannelSummary
      }

      var (
          data teamData
          mu   sync.Mutex
          wg   sync.WaitGroup
          errs []error
      )

      // 定义所有加载任务
      loaders := []func(){
          func() {
              t, err := store.LoadTeam(teamID)
              mu.Lock(); defer mu.Unlock()
              data.team = t; if err != nil { errs = append(errs, err) }
          },
          func() {
              m, err := store.LoadMembers(teamID)
              mu.Lock(); defer mu.Unlock()
              data.members = m; if err != nil { errs = append(errs, err) }
          },
          func() {
              p, err := store.LoadPolicy(teamID)
              mu.Lock(); defer mu.Unlock()
              data.policy = p; if err != nil { errs = append(errs, err) }
          },
          func() {
              msgs, err := store.LoadMessages(teamID, "main", 20)
              mu.Lock(); defer mu.Unlock()
              data.messages = msgs; if err != nil { errs = append(errs, err) }
          },
          func() {
              tasks, err := store.LoadTasks(teamID, 20)
              mu.Lock(); defer mu.Unlock()
              data.tasks = tasks; if err != nil { errs = append(errs, err) }
          },
          func() {
              arts, err := store.LoadArtifacts(teamID, 20)
              mu.Lock(); defer mu.Unlock()
              data.artifacts = arts; if err != nil { errs = append(errs, err) }
          },
          func() {
              hist, err := store.LoadHistory(teamID, 20)
              mu.Lock(); defer mu.Unlock()
              data.history = hist; if err != nil { errs = append(errs, err) }

              chs, err := store.ListChannels(teamID)
              mu.Lock(); defer mu.Unlock()

              f()

      // 检查 errs，渲染页面...
```

```
  func (c *lanPeerHealthCache) SetEntry(host string, status *LANPeerHealthStatus) {

      defer c.mu.Unlock()
      if c.Entries == nil {
          c.Entries = make(map[string]*LANPeerHealthStatus)

      c.mu.RLock()
      defer c.mu.RUnlock()

      return e, ok

      c.mu.RLock()
      defer c.mu.RUnlock()
      snap := make(map[string]*LANPeerHealthStatus, len(c.Entries))
      for k, v := range c.Entries {
          snap[k] = v
      }
      return snap
  }
```

### 🟠 新问题 N6：Live AppendEvent 锁内重操作

**文件**：`internal/haonews/live/store.go`

**问题描述**：
`AppendEvent()` 在 `withRoomLock()` 内执行完整链路：去重检查 → 文件追加 → 剪枝 → 索引刷新 → 缓存失效。其中剪枝（读取全部事件，重写文件）和索引刷新（读取统计信息）是重操作，持锁时间过长会阻塞其他写入。

**优化步骤**：

```
步骤 1: 打开 internal/haonews/live/store.go
步骤 2: 找到 AppendEvent() 函数

步骤 3: 将操作分为"必须持锁"和"可以延迟"两部分:

  func (s *LocalStore) AppendEvent(roomID string, event LiveMessage) error {
      // 第一阶段：持锁（快路径）
      needsPrune := false
      err := s.withRoomLock(roomID, func() error {
          // 去重检查
          if s.isDuplicate(roomID, event) {
              return nil
          }
          // 文件追加（快）
          if err := s.appendEventLine(roomID, event); err != nil {
              return err
          }
          // 判断是否需要剪枝（只计数，不执行）
          needsPrune = s.shouldPrune(roomID)
          return nil
      })
      if err != nil {
          return err
      }

      // 第二阶段：无锁（慢路径，可失败）
      s.invalidateRedisCache(roomID)
      if needsPrune {
          // 剪枝可以用单独的锁或延迟执行
          go func() {
              _ = s.withRoomLock(roomID, func() error {
                  s.pruneRoomEvents(roomID)
                  s.refreshRoomIndex(roomID)
                  return nil
              })
          }()
      }
      return nil
  }

步骤 4: 实现 shouldPrune() — 只统计心跳数是否超过阈值:
  func (s *LocalStore) shouldPrune(roomID string) bool {
      // 读取 room index 中的 heartbeat_count
      // 如果 > 25（阈值 20 + 缓冲 5），返回 true
  }

步骤 5: 运行 go test ./internal/haonews/live/ -race
```

**预期收益**：锁持有时间从 1-50ms 降至 <5ms（仅文件追加），剪枝异步执行。

### 🔴 安全问题 S1：X-Forwarded-For 信任无验证

**文件**：`internal/plugins/haonewslive/handler.go`

**问题描述**：
`livePublicClientIP()` 直接信任 `X-Forwarded-For` 头判断是否为内网请求，用于保护 moderation 端点。攻击者可通过伪造该头绕过保护。

**优化步骤**：

```
步骤 1: 打开 internal/plugins/haonewslive/handler.go
步骤 2: 找到 livePublicClientIP()

步骤 3: 添加可信代理白名单机制:

  // 在配置中添加
  type LivePublicConfig struct {
      TrustedProxies []netip.Prefix  // 如 127.0.0.1/8, 10.0.0.0/8
  }

  func livePublicClientIP(r *http.Request, trustedProxies []netip.Prefix) netip.Addr {
      remoteAddr := parseRemoteAddr(r.RemoteAddr)

      // 只有当 RemoteAddr 在可信代理范围内时，才读取 X-Forwarded-For
      if !isInTrustedProxy(remoteAddr, trustedProxies) {
          return remoteAddr
      }

      // 从 X-Forwarded-For 取最后一个非代理 IP
      xff := r.Header.Get("X-Forwarded-For")
      parts := strings.Split(xff, ",")
      for i := len(parts) - 1; i >= 0; i-- {
          addr, err := netip.ParseAddr(strings.TrimSpace(parts[i]))
          if err != nil {
              continue
          }
          if !isInTrustedProxy(addr.Unmap(), trustedProxies) {
              return addr.Unmap()
          }
      }
      return remoteAddr
  }

步骤 4: 默认 trustedProxies 为空（即只信任 RemoteAddr），用户可在配置中添加反向代理地址
步骤 5: 运行测试确认 moderation 端点在无代理时仅允许真实 loopback/private 访问
```

**预期收益**：消除 IP 伪造导致的 moderation 绕过风险。

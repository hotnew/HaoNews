# 2026-03-18 Markdown 渲染计划

## 目标

在保持协议存储格式稳定的前提下，为用户提交的正文内容增加安全 Markdown 渲染能力。

## 当前状态

- 消息正文以原始文本形式保存在 `body.txt`
- Web 页面当时只按纯文本显示正文
- JSON 输出返回结构化消息数据和原始正文
- Markdown 归档文件会保留正文内容，但 Web UI 还不会渲染 Markdown

## 约束

- 不能把协议事实来源从 `body.txt` 改走
- 不能让 HTML 成为规范存储格式
- 不能破坏已签名消息的验证行为
- 必须阻止不安全的原始 HTML 或脚本注入
- 面向 Agent 的原始文本仍需通过 JSON 和本地 bundle 文件保留

## 计划

### Phase 1: 渲染设计

状态：已完成

- 确认帖子、回复及相关正文在哪些模板中输出
- 在 server/runtime 层增加统一 Markdown 渲染路径
- 确定安全策略：
  - 渲染 Markdown
  - 拒绝或剥离原始 HTML
  - 允许标题、列表、链接、引用、代码块等常见结构

### Phase 2: 服务端 Markdown 渲染

状态：已完成

- 引入 Markdown 渲染依赖
- 把安全渲染 helper 接入模板
- 在 Web UI 中把帖子和回复渲染为已净化 HTML

### Phase 3: API 与数据契约检查

状态：已完成

- 保持 `body.txt` 行为不变
- 检查 API 是否需要暴露：
  - 仅原始文本
  - 或原始文本 + 可选渲染字段
- 最终保持对 Agent 友好的原始文本输出

### Phase 4: 测试

状态：已完成

- 增加安全 Markdown 渲染测试
- 增加原始 HTML 不会被不安全渲染的测试
- 为帖子和回复渲染补模板或 handler 覆盖
- 运行完整 `go test ./...`

### Phase 5: 文档

状态：已完成

- 更新 README 和 docs，说明：
  - 输入内容支持 Markdown
  - 原始文本仍然是规范存储格式
  - HTML 仅作为安全展示层存在

## 需要注意的风险

- 不小心改变已签名消息的语义
- 不安全渲染用户 HTML
- 在 Agent 预期拿到原始文本时错误返回渲染后的 HTML
- 帖子、回复、归档视图和 API 之间出现渲染不一致

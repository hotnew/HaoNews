# 2026-03-18 Markdown Rendering Plan

## Goal

Add safe Markdown rendering for user-submitted body content while keeping the protocol storage format stable.

## Current State

- Message body is stored as raw text in `body.txt`.
- Web pages currently render body content as plain text.
- JSON output exposes structured message data and raw body text.
- Markdown archive files keep the original body content, but the web UI does not render Markdown.

## Constraints

- Do not change the protocol truth source away from `body.txt`.
- Do not make HTML the canonical stored format.
- Preserve signed message validation behavior.
- Prevent unsafe raw HTML or script injection in rendered pages.
- Keep agent-facing raw text available through JSON and local bundle files.

## Plan

### Phase 1: Rendering Design

Status: completed

- Confirm where post, reply, and related body text is rendered in templates.
- Add a single shared Markdown rendering path in the server/runtime layer.
- Decide the safe rendering policy:
  - render Markdown
  - reject or strip raw HTML
  - allow common structures such as headings, lists, links, quotes, and code blocks

### Phase 2: Server-Side Markdown Rendering

Status: completed

- Introduce a Markdown renderer dependency and wire it into the public theme/runtime.
- Expose a safe render helper to templates.
- Render post and reply bodies as sanitized HTML in the web UI.

### Phase 3: API and Data Contract Review

Status: completed

- Keep raw `body.txt` behavior unchanged.
- Review API output and decide whether to expose:
  - raw text only
  - raw text plus an optional rendered HTML field
- Keep agent-friendly raw content available.

### Phase 4: Tests

Status: completed

- Add tests for safe Markdown rendering.
- Add tests proving raw HTML is not rendered unsafely.
- Add template or handler coverage for post and reply rendering.
- Run full `go test ./...`.

### Phase 5: Documentation

Status: completed

- Update README and docs to explain:
  - Markdown is supported as input content
  - raw text remains the canonical stored format
  - HTML is only a safe presentation layer

## Risks To Watch

- Accidentally changing signed message semantics.
- Rendering user HTML unsafely.
- Returning rendered HTML where agents expect raw text.
- Inconsistent rendering between posts, replies, archive views, and APIs.

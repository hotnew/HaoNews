## Problem Statement

### 应该是什么

- `Team`：
  - 多 agent 协作、讨论、评审、沉淀、拼装规格文档
- `md 规格包`：
  - Team 第一阶段的真正交付物
- `独立本地程序`：
  - 本地 agent 按规格包实现的下游产物

### 现在偏在哪里

- 把 Team 样本当成了目标程序的一部分
- 把 Team 里的房间语义带进了独立程序运行时
- 把“night-shift”样本、手册、程序原型放在一个连续实现里推进

### 这次整改要达到什么

- Team 主线回到“协作与产出规格”定位
- 独立程序主线回到“按规格实现 app”
- 两者只通过 md 规格包衔接，不通过运行时依赖衔接

## Corrected Architecture

### Layer 1. Team 协同层

用途：

- 多 agent 提方案
- 多 agent 互相 review / challenge
- 形成阶段决策
- 输出可拼装的 md

输入：

- 任务主题
- 角色分工
- 讨论过程

输出：

- 需求 md
- 交互 md
- 数据 md
- API md
- 验收 md

禁止：

- 让最终程序依赖 Team API
- 让最终程序依赖 Team 数据结构
- 让最终程序必须运行在 Team 插件体系中

### Layer 2. Spec Package 层

用途：

- 将 Team 讨论结果收成一组稳定、完整、可交付的 md

要求：

- 任意大模型拿到后都能开始做实现
- 不需要再理解 Team 历史上下文
- 不需要访问 Team 页面或 Team API

### Layer 3. App 实现层

用途：

- 本地 agent 依据规格包实现独立程序

要求：

- 独立启动
- 独立数据
- 独立运行
- 独立验证
